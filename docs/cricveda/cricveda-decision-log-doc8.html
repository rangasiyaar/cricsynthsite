Purpose

This is a LIVING DOCUMENT. Every significant decision made during CricVeda's development is recorded here.

Each entry records
  • WHAT was decided
  • WHEN it was decided
  • WHY that choice was made
  • WHAT alternatives were considered
  • WHAT trade-offs were accepted
  • WHAT would trigger revisiting this decision
  • STATUS: Active / Superseded / Revisit
Why this matters
  • Future-you forgets why past-you made a choice
  • New team members need context
  • Prevents re-debating settled decisions
  • Creates accountability for outcomes
  • Makes it easy to identify which decisions to reverse when circumstances change

Section 03

Scope Decisions

DEC-SCOPE-001 T20-Only Format Focus ✅ Active · March 2026
CricVeda covers T20 cricket only. No ODI, no Test, no T10. All code, schema, and algorithms optimized for T20.
Alternatives Considered
Option A: All formats (T20 + ODI + Test)Rejected
  • Larger addressable market
  • Year-round relevance (Tests fill T20 gaps)
  • 3× complexity in algorithms (T20 form ≠ Test form)
  • Schema must handle 5-day matches
  • Fantasy scoring rules differ per format
  • Phase definitions differ (PP=10 in ODI vs 6 in T20)
  • Massively increases MVP scope
Too much complexity for a solo builder.
Option B: IPL onlyRejected
  • Simplest scope (250 players, 74 matches)
  • 10-month dead period per year
  • Cross-league form impossible (it IS IPL only)
  • Debutant problem unsolvable
  • Competitive disadvantage (others cover IPL too)
Cross-league form is our core differentiator. IPL-only kills the unique value proposition.
Option C: T20 only (all leagues)Selected ✦
  • 70–80% of fantasy cricket activity
  • Cross-league form scoring possible
  • Year-round coverage
  • CricSheet has best T20 coverage
  • Single phase definition works across ALL T20 leagues
  • Misses ODI World Cup moments
  • Some domestic tournaments have sparse data
Trade-offs Accepted
  • Miss ODI/Test audience (small in fantasy context)
  • Cannot serve users during pure ODI/Test series
  • T10/Hundred variants treated as T20 (close enough in format)
Revisit When
  • Users explicitly request ODI/Test support
  • Revenue exceeds $5,000/month (can invest in multi-format development)
  • ODI World Cup drives significant demand
DEC-SCOPE-002 All T20 Leagues, Not IPL-Only ✅ Active · March 2026
Cover ALL T20 leagues and internationals globally. API serves all leagues. Analytics engine processes all. Not limited to IPL or any single league.

The original consideration was IPL-only for simplicity. Discussion revealed two fatal flaws: (1) form scores based on 10-month-old IPL data are misleading — players play other leagues between seasons, and (2) debutants/returnees have "no data" which is unacceptable for ₹20+ crore players. The insight: the API is IPL-focused but the analytics engine must see ALL T20 data for accurate form scoring.

Key Alternatives
Option B: IPL-focused with global form engineRejected
  • Simpler API surface
  • Wastes cross-league data already ingested
  • Multi-league users (Persona Sam) not served
  • 10-month dead period for API revenue
If we're ingesting all data anyway, why artificially limit the API to one league?
Option C: All T20 leaguesSelected ✦
  • Same ingestion pipeline serves all leagues
  • League-agnostic architecture (no per-league code)
  • Year-round revenue
  • Cross-league form = core differentiator
  • More fixtures and squads to manage manually
  • Some leagues have less user demand
Trade-offs Accepted
  • Higher manual effort for fixtures across leagues
  • Metadata coverage lower for obscure league players
  • Cannot deeply serve ALL leagues equally — tier coverage: Tier 1 full, Tier 2 standard, Tier 3 data-only for form engine
Revisit When
  • Operator can't maintain multiple leagues alone → prioritize top 3–5
  • User demand data shows 90%+ is IPL only → reconsider focus
  • Storage limits hit from too many leagues
DEC-SCOPE-003 API-First, Dashboard Second Active
Build the API first. The dashboard consumes the API. API is the product. Dashboard is a client of the product.
Dashboard-first has no B2B revenue path and no developer ecosystem. Developers can't build on top of it. API-first enables platform integrations, creator tools, and developer ecosystem. Dashboard is just one client of many — forces clean separation of data and presentation.
Trade-off: Dashboard development is constrained by API design (good — forces good API design). Dashboard ships in same timeline (Days 11–13) using its own API.
Revisit: If 90%+ users only use dashboard and nobody uses API, or B2B revenue is $0 after 6 months.
DEC-SCOPE-004 Fantasy-Focused, Not General Cricket Analytics Active
All features, algorithms, and endpoints are designed for fantasy cricket decision-making. Form scores optimized for fantasy relevance. Recommendations framed as "pick this player."
General cricket analytics has no clear monetization (who pays for general stats when ESPNcricinfo is free?). Fantasy users have clear willingness to pay for edge. B2B opportunity with fantasy platforms exists.
Trade-off: Excludes pure cricket research use case. Tied to fantasy sports regulation. But underlying data is general — non-fantasy users CAN use form scores and matchups.
Revisit: If fantasy sports banned in key markets → pivot to general analytics. If significant demand from non-fantasy users.
DEC-SCOPE-005 Pre-Match Intelligence, Not Live Active
CricVeda provides PRE-MATCH intelligence only. No live scoring, no in-match updates, no live ball-by-ball feed.
Fantasy teams are locked BEFORE the match starts — all value is in pre-match decisions. Live data requires paid feeds ($500+/month) and WebSocket infrastructure. Cricbuzz/ESPNcricinfo already do live scoring well — don't compete where established players dominate.
Trade-off: Cannot offer live win probability or live fantasy points tracking. Post-match data arrives via CricSheet (delayed).
Revisit: When revenue allows paid live data feed, or user demand for live features is overwhelming.
DEC-SCOPE-006 Men's Cricket Only for MVP Active
MVP covers men's T20 cricket only. Women's T20 is a future expansion.
CricSheet men's T20 data is most comprehensive. Fantasy cricket for men's leagues is 95%+ of market. Women's T20 fantasy is growing but small currently.
Trade-off: Miss growing WPL market. Miss Women's T20 World Cup moments. Perception risk.
Revisit: When WPL or Women's T20I fantasy demand is demonstrated. Target: add women's T20 by end of Year 1.
Section 04

Data Decisions

DEC-DATA-001 CricSheet as Primary Data Source ✅ Active · March 2026
CricSheet is the primary and foundational data source for all ball-by-ball match data.
Alternatives Considered
Option A: Paid data API (SportRadar, Roanuz)Rejected
  • Real-time, comprehensive, reliable
  • Player metadata included
  • $500–5,000/month — violates $0 budget constraint
  • Vendor lock-in
Cost prohibitive for MVP.
Option B: ESPN/Cricbuzz scrapingRejected
  • Comprehensive data available
  • Free
  • Explicitly prohibited by Terms of Service
  • Legal risk for commercial product
  • Fragile (site changes break scrapers)
  • IP bans, CAPTCHAs, maintenance nightmare
Legal risk + fragility. "Scrapers are sexy. Manual CSVs are boring. Boring ships."
Option C: CricSheetSelected ✦
  • 100% free · CC BY 4.0 (legal for commercial use)
  • Ball-by-ball granularity · ~4,500 T20 matches
  • Cross-league player IDs (ESPNcricinfo registry)
  • Structured JSON · Community-maintained, transparent
  • No player metadata (style, role, country)
  • No upcoming fixtures · No playing XI data
  • Update delay (not real-time)
  • Community project (single maintainer risk)
How We Handle the Cons
  • Metadata: 4-layer enrichment (DEC-DATA-003)
  • Fixtures: Manual CSV upload (DEC-DATA-004)
  • Playing XI: Manual admin updates (DEC-DATA-005)
  • Delay: Pre-match focus makes delay acceptable
  • Risk: Archive data locally, monitor for changes
Dependency Risk Mitigation
  • Archive all CricSheet data locally on each download
  • CC BY 4.0 is irrevocable for released data
  • Parser handles format variations gracefully
  • Long-term: diversify to CricAPI/Roanuz if revenue supports
  • Contribute back to CricSheet community
Revisit When
  • CricSheet stops updating for >30 days
  • CricSheet changes license terms
  • Revenue exceeds $500/month (can afford CricAPI Pro)
  • Real-time data becomes a user requirement
DEC-DATA-002 Cross-League Form Scoring ✅ Active · March 2026
Player form scores are calculated using weighted data from ALL T20 leagues, not just the league being queried. This IS the core differentiator — no competitor does this.
Weight Table
Same league (e.g., IPL for IPL query)35%
T20 Internationals25%
Other franchise T20 leagues20%
Career stats in queried league10%
Venue-specific history10%
Alternatives Considered
Option A: Single-league form onlyRejected
  • Produces misleading scores (SKY IPL vs SA20 example)
  • "No data" for debutants
  • No differentiation from existing platforms
Kills the unique value proposition.
Option B: Equal weighting across all leaguesRejected
  • IPL form SHOULD matter more for IPL query
  • BBL conditions ≠ IPL conditions
  • Different quality of opposition
Not all T20 leagues are equal.
Trade-offs Accepted
  • More complex algorithm to build and test
  • Weights are initially hand-tuned (not ML-optimized)
  • Users may question why non-IPL data affects score → solved by transparency (show data sources)
  • Weights need season-over-season tuning based on accuracy data
Revisit When
  • First full season of accuracy data available → tune weights based on prediction vs actual
  • ML model can learn optimal weights from historical data
  • Users consistently say cross-league data is noise
DEC-DATA-003 4-Layer Metadata Enrichment ✅ Active · March 2026
Player metadata (country, role, batting/bowling style) is enriched using 4 automated layers — NOT manual entry for 3,000+ players.
4 Layers (in order)
  • Layer 1: Derive from CricSheet match data (country from T20I team, role from batting/bowling ball counts)
  • Layer 2: Wikidata SPARQL API (batting/bowling style, DOB) — matches by ESPNcricinfo ID
  • Layer 3: CricAPI free tier batch fill (100/day for gaps)
  • Layer 4: Smart defaults (right-hand batting default = 75% of cricketers)
  • Override: Admin manual entry (tagged "manual", overrides all automated sources)
Coverage Targets
  • 100% of players have role derived
  • 85%+ have country assigned
  • 70%+ have batting/bowling style
  • Every field has source tag + confidence level
  • Sources: confirmed / wikidata / derived / cricapi / inferred / default / manual
Trade-offs Accepted
  • ~10% of players have partial/estimated metadata
  • CricAPI batch takes 5 days (100/day limit)
  • Some "default" values will be wrong — all mitigated by source tagging and confidence scores
DEC-DATA-004 Manual Fixtures, Not Scraped Active
Match schedules uploaded via CSV, not scraped from any website.
Legal (no scraping), reliable (100% accurate), manageable (~74 IPL matches = 30 min). CricAPI free tier as backup for fixtures.
Trade-off: Manual effort per season (~1–2 hours per league).
Revisit: When revenue allows CricAPI paid tier for auto-fixtures.
DEC-DATA-005 Manual Playing XI, Not Automated Active
Playing XI updated manually by operator via admin panel. Predicted XI as fallback.
No free, reliable, legal source for real-time XI. Manual update takes 2–3 min.
Trade-off: Operator must be available ~30 min before each match. Stale XI if operator is unavailable.
Revisit: When revenue allows CricAPI paid tier or B2B partner provides XI data.
DEC-DATA-006 No ESPN/Cricbuzz Scraping Permanent
CricVeda will NOT scrape ESPNcricinfo, Cricbuzz, or any site that prohibits scraping in ToS.
Legal risk (commercial product built on scraped data = lawsuit potential). Fragility (scrapers break). Maintenance burden (weekends fixing scrapers). "Boring ships. Sexy breaks at 2 AM."
Trade-off: More manual work for data that could be scraped. Some data unavailable (pitch reports, injury news).
Revisit: Never. This is a permanent ethical/legal decision.
DEC-DATA-007 Confidence Scoring on All Outputs Active
Every analytical output includes a confidence score (0.0–1.0), data source list, and data depth label.
Transparency builds trust. Users must know when data is limited. Distinguishes CricVeda from black-box tips. "Honest uncertainty" is a core principle.
Trade-off: More complex response format. Some users may find confidence scores confusing.
Revisit: If user feedback says confidence scores are confusing → simplify to 🟢🟡🔴 visual indicators.
DEC-DATA-008 Smart Defaults Over Missing Data Active
When metadata is missing, use educated defaults (e.g., right-handed batting default) rather than returning null.
"Something useful with a warning" is better than "nothing." 75% of batters are right-handed (safe default). Flagged as "default" source with "low" confidence.
Trade-off: Some defaults will be wrong. But clearly labeled as defaults, not stated as facts.
Revisit: If users report being misled by default values → consider returning null instead.
DEC-DATA-009 Pre-Computation Over Real-Time Queries Active
Compute all analytics via cron jobs. API serves cached results. No heavy computation on the request path.
Free tier limits (Vercel, Supabase) cannot handle complex queries per request at scale. Pre-computation moves work to GitHub Actions cron (free, async). API response time <500ms because it's just a cache lookup.
Trade-off: Data can be up to 24 hours stale for form scores. Match insights up to 1 hour stale. Recomputation needed when XI changes.
Revisit: When paid infrastructure allows real-time computation OR caching proves insufficient.
DEC-DATA-010 League-Agnostic Parser Architecture Active
The CricSheet parser and all analytics algorithms are league-agnostic. No "if IPL then..." code. League is a parameter, not a code path.
CricVeda covers 10+ leagues. Per-league code = 10× maintenance. New leagues should require config, not code. CricSheet format is consistent across leagues.
Trade-off: Cannot optimize for league-specific quirks. Hundred's 100-ball format handled as configuration exception.
Revisit: If specific leagues have fundamentally different data structures requiring separate handling.
Section 05

Tech Decisions

DEC-TECH-001 Vercel All-In-One ✅ Active · March 2026
Host BOTH the dashboard AND the API on Vercel using Next.js. No separate backend server.
Alternatives Considered
Option A: Render (API) + Vercel (Dashboard)Rejected
  • Python backend (FastAPI) for analytics
  • Render free tier spins down after 15 min
  • 30–50 second cold start (unacceptable)
  • Only ~500 MAU capacity
  • Two services to manage
  • 512 MB RAM, 0.1 CPU on free tier
Cold start is unacceptable. Capacity too low.
Option B: Vercel all-in-oneSelected ✦
  • 100K serverless function calls/month free
  • No cold start (serverless, always ready)
  • Edge network (fast globally)
  • One deployment — Next.js handles everything
  • ~10,000 MAU capacity on free tier
  • Automatic HTTPS, CDN, preview deployments
  • Backend logic in TypeScript, not Python
  • Serverless function 10s timeout on free tier
  • No persistent server (must be stateless)
Option C: Cloudflare Workers + VercelRejected
  • 100K requests/DAY free · zero cold start · ~50K MAU
  • Two platforms to manage
  • Workers = 10ms CPU time limit
  • Overkill for MVP
Added complexity not needed at MVP scale. Revisit when Vercel limits are hit.
Trade-offs Accepted
  • TypeScript instead of Python for analytics code (acceptable — TS handles math fine)
  • Serverless = stateless (no in-memory caching, use Redis instead)
  • 10-second function timeout (pre-computation avoids this on request path)
  • Vendor lock-in to Vercel deployment model
Revisit When
  • MAU exceeds 10,000 → need Vercel Pro at $20/month
  • Function timeout is hit on complex endpoints
  • Need persistent WebSocket connections (live features)
  • Python-specific ML libraries needed → add separate Python service at that point
DEC-TECH-002 Next.js with App Router Active
Use Next.js (latest) with App Router for the entire application — dashboard + API routes.
Industry standard for React + serverless. App Router is the future of Next.js. API routes replace separate backend. ISR for cached pages. Server Components reduce client JS.
Alternatives rejected: Plain React + separate API (two codebases), Remix (less ecosystem), Astro (limited API route support).
DEC-TECH-003 Supabase for Database + Auth Active
Use Supabase for PostgreSQL database AND authentication (email + password).
Free tier = 500 MB storage, 50K MAU auth, built-in auth UI helpers, Row Level Security, connection pooling.
Trade-off: Vendor lock-in to Supabase. Pauses after 1 week inactivity (mitigated by cron pings).
Alternatives rejected: Firebase (NoSQL not ideal for relational cricket data), PlanetScale (MySQL, less mature free tier), self-hosted Postgres ($0 budget).
DEC-TECH-004 Upstash Redis for Cache + Rate Limiting Active
Use Upstash Redis for API response caching and rate limiting.
Free tier = 10K commands/day. Works on Vercel Edge. @upstash/ratelimit handles rate limiting. Serverless-compatible (HTTP-based Redis, not TCP).
Trade-off: 10K commands/day limit. Rate limiting costs 2 commands per request. Must be efficient with cache reads.
DEC-TECH-005 GitHub Actions for Cron Jobs Active
Use GitHub Actions for all scheduled tasks (data ingestion, pre-computation, enrichment).
Free = 2,000 min/month. Can run Python or Node scripts. Triggered by cron or manual dispatch. Already using GitHub for version control.
Trade-off: 2,000 min/month limit. Must keep jobs efficient. Cron minimum interval = 5 minutes.
DEC-TECH-006 TypeScript for Entire Codebase Active
TypeScript (not JavaScript, not Python) for all application code including analytics.
Type safety prevents bugs in complex analytics code. Next.js has first-class TypeScript support. Single language for frontend + backend + scripts. Better IDE support.
Trade-off: Some cricket analytics libraries are Python-only — not a blocker since algorithms are custom anyway.
DEC-TECH-007 No Separate Backend Active
No FastAPI, no Express, no separate backend service. All backend logic lives in Next.js API routes.
One codebase. One deployment. One monitoring point. Pre-computation in cron jobs means API routes are simple (read from cache/DB, format response, return).
Trade-off: Cannot use Python libraries in API routes. Cron scripts CAN be Python (GitHub Actions) but API routes must be TypeScript.
DEC-TECH-008 PostgreSQL Over SQLite for Production Active
Use PostgreSQL (via Supabase) for production, not SQLite.
Concurrent access (multiple serverless functions reading simultaneously). Full-text search capability. JSONB support for match insights. Better indexing.
Trade-off: Cannot run database locally without Supabase CLI or Docker. Remote database = network latency on every query (mitigated by caching).
Section 06

API Decisions

DEC-API-001REST Over GraphQLActive
RESTful JSON API, not GraphQL.
Simpler to implement and document. Fantasy users and small platforms are familiar with REST. URL-based caching is straightforward. GraphQL adds complexity without clear benefit for 14 endpoints with predictable response shapes.
Trade-off: Multiple API calls for complex data needs (vs one GraphQL query) — mitigated by rich response objects that combine related data.
DEC-API-002Self-Built API Key AuthActive
Build API key generation, hashing, and validation in-house. Store key hashes in Supabase api_keys table. Format: cv_live_xxxxxxxxxxxxxxxxxxxx.
Full control. Unlimited keys. No external dependency. Simple enough to implement (SHA-256 hash + DB lookup). No cost.
Alternative Unkey.dev rejected: 1,000 free verifications/month limit. Trade-off: ~2–3 hours of implementation work.
DEC-API-003Daily Rate Limits, Not Per-SecondActive
Rate limits are DAILY (100 calls/day free, 5,000/day pro) with a per-second burst limit of 10.
"You have 100 API calls per day" is clearer than per-minute limits. Fantasy usage is bursty (all calls before one match) not sustained — daily limits fit the pattern. Redis commands saved.
Trade-off: User could burn 100 calls in 10 seconds (allowed but unusual). Burst limit of 10/sec prevents accidental loops.
DEC-API-004URL-Path VersioningActive
API version in URL path: /api/v1/players/{id}/form
Most intuitive versioning method. Easy to implement with Next.js folder structure. Clear in documentation and logs.
Trade-off: URL changes between versions. Mitigated by supporting v1 indefinitely when v2 launches.
DEC-API-00514 Endpoints for MVP (Not 18)Active
Ship 14 of 18 planned endpoints. Deferred: vs-bowler-type, key-battles, differentials, venues/{id}.
14 endpoints cover all core user needs. Deferred endpoints need higher metadata coverage or more complex algorithms. Core value delivered without them.
Trade-off: Some docs say "coming soon" for 4 endpoints. Power users may want them sooner.
Revisit: Week 3 post-launch — add all 4 deferred endpoints.
DEC-API-006Free Tier Includes Basic EndpointsActive
Free = player stats, form scores, match listings, insights (basic), matchups (top 3), playing XI. Pro adds dream team, captain picks, differentials, full matchup depth.
Free tier must be USEFUL enough to hook users. Balance: free = data and analysis; pro = recommendations and actionable picks. Form scores attract users who then want captain picks (Pro).
Trade-off: Giving away form scores for free. But form scores ATTRACT users who then upgrade for captain picks.
DEC-API-007Swagger/OpenAPI for DocumentationActive
Auto-generate API documentation from OpenAPI spec using Swagger UI at /docs.
Write once (OpenAPI spec), get interactive docs + try-it-live for free. Industry standard. Developers expect it. Reduces documentation maintenance burden.
Trade-off: Swagger UI looks generic. Not as pretty as custom-designed docs. Acceptable for MVP.
Section 07

Business Decisions

DEC-BIZ-001Zero-Cost Infrastructure at LaunchActive
All infrastructure must cost $0/month at launch and through Year 1 unless revenue justifies upgrades.
No external funding. No revenue yet. Product must be viable at $0 to validate before investing. Free tiers of modern cloud services are generous enough. Forces architectural efficiency.
Upgrade triggers: Vercel Pro ($20/mo) at 8K MAU · Supabase Pro ($25/mo) at 400MB · Upstash Pro ($10/mo) at 8K commands/day · CricAPI Pro ($30/mo) at $500/mo revenue. Only exception: domain name (~$10/year).
DEC-BIZ-002Freemium Pricing ModelActive
Three tiers — Free, Pro ($9/month), Business ($49/month).
Freemium is proven for API products (Stripe, Twilio, OpenWeatherMap). Free drives adoption. Pro monetizes power users and creators. Business monetizes B2B platform integrations.
Alternatives rejected: fully paid (blocks adoption), fully free with ads (no infrastructure), usage-based pricing (too complex for MVP).
DEC-BIZ-003$9/Month Pro Tier PricingActive
Pro tier priced at $9/month ($7/month if annual).
Benchmarked: CricAPI Pro is $30/month (we provide more — intelligence, not just data). $9/month accessible for Indian users (₹750/month). Fantasy Telegram groups charge ₹500–5,000/month for worse quality tips.
Trade-off: May be too expensive for casual Indian users → consider India micro-tier later (₹99/month).
Revisit: After 100 paid users — analyze conversion rate, churn, and feedback on pricing.
DEC-BIZ-004Manual Pro Upgrades for MVPActive
No Stripe in MVP. Users who want Pro contact via email. Admin manually upgrades their plan in the database. Payment collected via UPI/PayPal.
Stripe integration takes 1–2 days. Expected Pro users in Month 1: <20. Manual handling of <20 upgrades is trivial. Shipping faster is more important than payment automation.
Trade-off: Some users won't upgrade due to friction. Not scalable beyond ~50 users.
Revisit: When Pro user count exceeds 20 OR manual upgrades take more than 30 min/week → build Stripe integration.
DEC-BIZ-005B2B as Primary Revenue TargetActive
Fantasy platforms (B2B) are the primary revenue target, not individual users (B2C).
1 B2B customer at $49/month = 5 Pro B2C customers at $9/month. B2B has higher retention (integration lock-in). B2B scales CricVeda's reach (1 platform = thousands of end users).
Trade-off: Must invest in B2B materials (integration guide, SLA, support channel). B2B sales cycle is longer.
Revisit: If B2B revenue is $0 after 6 months → pivot to B2C focus.
DEC-BIZ-006Creator Partnerships for DistributionActive
Partner with fantasy cricket YouTube/Twitter creators. Provide Pro access free or discounted. They mention CricVeda in content = free marketing.
Zero marketing budget. Creators have 10K–500K+ fantasy cricket fans. One mention = thousands of potential signups. Creator cost = $0–9/month. Marketing equivalent value = $100–10,000.
Trade-off: Dependent on creator willingness. Cannot control their message. Must select creators carefully.
Revisit: If creator mentions don't drive signups → try other channels (Reddit, HN, Product Hunt, SEO).
DEC-BIZ-007CricSheet CC BY 4.0 AttributionPermanent
CricSheet attribution displayed on every page that uses CricSheet-derived data, in footer and on data methodology page. License compliance is non-negotiable.
Legal requirement of CC BY 4.0. Also good practice to credit open-source data providers. Builds trust with technically savvy users.
Attribution text: "Ball-by-ball data sourced from CricSheet (cricsheet.org), licensed under CC BY 4.0." Displayed: landing page footer, match detail footer, API docs, methodology page, README.
Revisit: Never. Permanent legal and ethical requirement.
Section 08

Operations Decisions

DEC-OPS-001Solo Operator Model for MVPActive
One person operates the entire system for MVP. All manual tasks designed for solo execution.
No team. No budget for team. Must be sustainable for one person. Total match-day work: <5 min per match. Total per-season setup: <8 hours per league.
Trade-off: Single point of failure. If operator is sick/busy, predicted XI fallback exists. Cannot scale to 4+ concurrent leagues without significant time investment.
Revisit: When revenue allows hiring part-time help OR community volunteers can assist with data.
DEC-OPS-002<5 Min Match-Day Operations TargetActive
All match-day admin tasks must complete in under 5 minutes per match.
Sustainability. IPL has 74 matches in 2 months, sometimes 2/day. If match-day ops take 30 min, operator burns out by match 10. Breakdown: check XI (1 min) + update admin (2 min) + verify recompute (1 min) + check page (1 min) = 5 min.
Trade-off: Admin panel must be very efficient (dropdowns, checkbox XI, one-click recompute). No time for complex workflows.
Revisit: If admin panel is slow → optimize UI. If more operations added → reassess time budget.
DEC-OPS-003CSV-Based Fixture ManagementActive
Match schedules loaded via CSV file upload. CSV format: date, time, team1, team2, venue, league, season.
IPL schedule = 74 matches. Entering one by one = tedious. CSV upload = paste from Google Sheets in 5 minutes. Same approach works for all leagues.
Trade-off: Must prepare CSV externally. CSV format must be validated carefully.
Revisit: If admin panel is frequently used for fixtures → build richer fixture management UI.
DEC-OPS-004Predicted XI as Default FallbackActive
If no XI is manually set, use the most recent XI for that team (from their last match) as a "predicted" XI. Users clearly see ⚠️ Predicted vs ✅ Confirmed.
Teams rarely change more than 1–2 players between matches (85–90% accuracy). Better than showing "no XI available." Ensures system always has some XI to work with.
Trade-off: Predicted XI is sometimes wrong (1–2 players off). Dream team may include a player who isn't playing. Mitigated by clear labeling.
Revisit: If predicted XI accuracy drops below 80%. If automated XI source becomes available → switch.
