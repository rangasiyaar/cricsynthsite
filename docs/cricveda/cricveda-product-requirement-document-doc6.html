Section 06 — Feature Group F1

Data Pipeline Requirements

F1.1 CricSheet Ingestion
Parse ball-by-ball match data from CricSheet for ALL T20 formats and leagues.
Key Functional Requirements
  • Parse all CricSheet JSON T20 match files (t20s_json.zip + league zips)
  • Extract and deduplicate players using CricSheet people registry (ESPNcricinfo IDs)
  • Store every delivery as a row: batter, bowler, runs, extras, wicket info
  • Support incremental ingestion — only process new/updated matches on re-run
  • Idempotent: re-running doesn't create duplicates
  • League-agnostic parsing — same parser handles ALL T20 leagues, no league-specific code
  • Handle all CricSheet JSON schema variations gracefully (log warnings, don't crash)
Performance Targets
  • Initial bulk load: ~4,500 T20 matches in <30 min
  • Incremental update: new matches in <5 min
  • Trigger: manual or via cron (GitHub Actions)
Acceptance Criteria
AC-1All ~4,500 matches parsed without error
AC-2All players have unique IDs matching registry
AC-3Delivery count matches source (spot-check 10 matches)
AC-4Re-running doesn't create duplicate records
AC-5New league zip produces correct data without code change
F1.2 Player Metadata Enrichment
Enrich player records with country, role, batting style, bowling style, and bowling type using a 4-layer automated pipeline plus manual override.
4-Layer Enrichment Pipeline
  • Layer 1 — Match data: Derive country from T20I matches, role from balls batted/bowled. Coverage: 100% role, 80–85% country.
  • Layer 2 — Wikidata SPARQL: Query all cricketers by ESPNcricinfo ID. Batting style, bowling style, DOB. Coverage: 65–75%.
  • Layer 3 — CricAPI free tier: Gap-fill batch (100 req/day). Prioritize by match count. Auto-run via cron.
  • Layer 4 — Smart defaults: Batting default "right" (75% of cricketers). Remaining 5–10% of players.
  • Manual override: Admin can override any field at any time — tagged "manual", overrides all automated sources.
Source Tagging (Every Field)
  • Every metadata field has: source tag + confidence level
  • Sources: "confirmed" / "wikidata" / "derived" / "cricapi" / "inferred" / "default" / "manual"
  • Confidence: "high" / "medium" / "low"
  • Surfaced in API responses to inform users of data quality
Coverage Targets
Target90%+ players have country assigned
Target100% players have role derived
Target70%+ have batting and bowling style
F1.4 Playing XI Management
Support managing playing XI for upcoming matches — both predicted and confirmed — with automatic recomputation on update.
Requirements
  • Status: "predicted" or "confirmed" — clearly distinguished in API
  • Default predicted: use last match's XI for that team
  • Admin UI: select match → select team → pick 11 from squad in <2 min
  • On save: recompute dream team, captain picks, key battles, differentials
  • Recomputation completes in <60 seconds
Recomputation Triggers
  • XI confirmed or changed → recompute all match insights
  • Recomputation: <60 seconds end-to-end
  • New insights immediately available via API
AC-1Admin sets both teams' XI in <3 min total
AC-4XI update triggers recomputation within 60 seconds
F1.5 Pre-Computation Engine
Pre-compute all analytics results and cache them. API requests serve cached data — NOT real-time queries. This is the scalability strategy.
Form Scores
Daily + on match completion. Redis TTL: 24 hours.
Match Insights
Daily + when XI confirmed. Redis TTL: 1 hour.
Fantasy Recs
When XI changes. Redis TTL: 30 minutes.
Venue Stats
After each match at venue. Redis TTL: 7 days.
Matchups
After each match. Min 6 balls. Redis TTL: 24 hours.
Section 07 — Feature Group F2

Analytics Engine Requirements

F2.1 Form Score Calculator
Calculate a 0–10 form score for any player, weighted across ALL T20 leagues and formats. This is CricVeda's core differentiator.

Cross-League Weighting

Data SourceWeight
Last 5 innings in SAME league as query (e.g., IPL for IPL query)35%
Last 5 T20I innings25%
Last 5 other franchise T20 innings20%
Career stats in queried league10%
Venue-specific history10%
Output Fields
  • form_score: decimal 0.0–10.0
  • trend: "rising" / "stable" / "declining"
  • confidence: decimal 0.0–1.0
  • consistency_index: decimal 0.0–1.0
  • last_5_scores: array of recent innings
  • data_sources: array of contributing leagues
  • data_depth: "strong" / "moderate" / "limited" / "minimal"
Edge Case Handling
  • New player (0 matches): form_score = null, estimated potential from archetype
  • <5 total T20 matches: confidence = "low", small sample flagged
  • Long absence (>6 months): pre-absence data with lower confidence
  • All edge cases return valid response — never null/crash
AC-5Calculation completes in <100ms per player
AC-6Bumrah in form = 8+, struggling player = 3–4
F2.2 Matchup Analyzer
Calculate batter-vs-bowler H2H statistics across ALL T20 leagues combined (not per-league).
Analysis Types
  • Batter vs Specific Bowler: balls, runs, dismissals, SR, dot %, dominance score (0–10). Min 6 balls.
  • Player vs Bowling Type: pace / spin / left-arm etc. across all T20 leagues.
  • Player vs Specific Team: all innings against that team across leagues.
  • Key Battles (for match): top 5 H2H pairs ranked by interactions × one-sidedness, with human-readable fantasy notes.
  • Team vs Team H2H: historical results within same league only.
Sample Key Battle Output
"Rohit vs Chahar"
balls: 28 · dismissals: 5
SR: 82 · dot_pct: 58%
advantage: bowler
fantasy_note: "Chahar owns Rohit
in powerplay — SR 82"
AC-4Insufficient data flagged for <6 balls
AC-6All calculations <200ms
F2.6 Dream Team Generator
Generate a recommended fantasy XI from both teams' playing XI, optimized for maximum expected fantasy points within constraints.
Constraints (All Must Be Met)
  • Exactly 11 players
  • 1–4 wicketkeepers
  • 3–6 batters
  • 1–4 all-rounders
  • 3–6 bowlers
  • Minimum 1 player from each team
  • Maximum 7 players from one team
  • Player roles from metadata
Output Format
  • Selected XI with role and reasoning per pick
  • Every player has 1–3 reasoning strings (specific data, not adjectives)
  • Total expected fantasy points for team
  • Confidence score for team as a whole
AC-1Dream team respects ALL constraints
AC-3Every player has at least 1 reasoning string
F2.7 Captain Picker
Recommend top 3 captain and vice-captain choices with detailed, data-specific reasoning.
Captain Score Weights
  • Form score — 35%
  • Venue history score — 25%
  • Favorable matchups in this match — 20%
  • Consistency index — 10%
  • Role ceiling (all-rounders bonus) — 10%
Anti-Pick Warning

If a popular likely captain has a BAD matchup, surface a warning automatically:

"Rohit likely high-ownership captain but Chahar has dismissed him 5 times (SR 82)"
AC-2Reasoning includes specific numbers, not adjectives
AC-3Risk varies across the 3 picks

Section 13

Constraints

Constraint 1
Zero Budget

All services within free tier. No paid subscriptions until product generates revenue. Domain (~$10/year) is the only acceptable expense.

Services: Vercel · Supabase · Upstash · GitHub Actions
Constraint 2
Solo Operator

All manual operations must be doable by one person in reasonable time. Match-day operations: <5 min/match. Per-season setup: <8 hours/league.

Constraint 3
Open Data Only

No scraping of ToS-protected websites. Only legally clean sources. Manual data entry where automated sources unavailable.

CricSheet (CC BY 4.0) · Wikidata (CC0) · OpenWeatherMap (free) · CricAPI (free tier)
Constraint 4
Free Tier Service Limits

Hard limits that shape architecture decisions.

Vercel: 100K req/mo · Supabase: 500MB DB · Upstash: 10K cmd/day · GitHub Actions: 2,000 min/mo · OWM: 1K calls/day · CricAPI: 100/day
Constraint 5
Pre-Computation Over Real-Time

API must serve pre-computed cached results, NOT run complex queries per request. Heavy computation happens in cron jobs, not the request path.

Constraint 6
T20 Format Only

No ODI or Test cricket. All code, schema, and algorithms optimized for T20. Format expansion is out of scope.

Constraint 7
League-Agnostic Architecture

No league-specific code. All leagues processed by the same parser, same algorithms, same endpoints. League is a filter parameter, not a separate code path.
