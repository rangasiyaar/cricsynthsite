Section 01

Purpose of This Document

The PRD (Document #06) defines EVERYTHING the product could do. This document defines what ships FIRST.

MVP = the smallest version of CricVeda that: solves a real problem for real users · is good enough to get honest feedback · proves (or disproves) our core assumptions · can be built by one person in ~2–3 weeks · costs $0 to run · is NOT embarrassing to show publicly.

This document makes HARD CUTS. If it's not listed here, it does NOT ship in MVP. No scope creep. No "just one more feature."

THE ENEMY OF SHIPPING IS "ALMOST DONE." MVP SHIPS INCOMPLETE BUT USEFUL.
Section 02

Core Assumptions We're Testing

The MVP exists to validate these 5 assumptions. If any of them are wrong, we learn faster by shipping than by continuing to build.

Assumption 1
Cross-league form scoring is valuable
"Fantasy users will find cross-league T20 form scores more useful than single-league stats."
How we test
  • Do users return for second match?
  • Do they share form scores?
  • Do creators cite form scores in content?
Killed if: Users say "I don't care about BBL form for IPL picks" or ignore form scores entirely.
Assumption 2
Matchup data drives fantasy decisions
"Users will change their fantasy picks based on specific matchup data they couldn't access before."
How we test
  • Is matchup endpoint among top 3 used?
  • Do users mention matchups in feedback?
Killed if: Nobody uses matchup endpoint, or data is too sparse to be useful.
Assumption 3
API delivery model works for cricket
"Developers and platforms will integrate a cricket intelligence API into their products."
How we test
  • Do developers sign up and make API calls?
  • Do any B2B inquiries come in within 3 months?
Killed if: Nobody uses the API — everyone prefers the dashboard. API-first model is wrong.
Assumption 4
Free CricSheet data is sufficient
"Ball-by-ball data from CricSheet, enriched with free metadata sources, produces insights accurate enough that users trust and act on them."
How we test
  • Algorithm accuracy tracking (predictions vs actuals)
  • User trust scores in feedback surveys
Killed if: Data gaps too large. Form scores wildly inaccurate. Users say "this data is wrong."
Assumption 5
Manual operations are sustainable
"One person can maintain playing XI updates, fixture management, and data quality for an active T20 league."
How we test
  • Operator burnout tracking · time per match
  • Missed updates during live league
Killed if: Operator can't keep up with match-day pace, data goes stale, users lose trust.

Section 10

MVP Technical Boundaries

The MVP CAN Handle
  • 5,000–10,000 MAU (within free tier limits)
  • 30–50 concurrent users (Vercel handles)
  • 100–200 API calls/day per free user
  • 14 endpoints serving cached data
  • <500ms response time for 95% of requests
  • ~4,500 T20 matches in database (~100–200 MB)
  • ~3,500–4,000 unique players
  • 10–20 active league schedules
  • Pre-computed insights for upcoming 7 days
  • 1 operator · 5–10 min per match day
The MVP CANNOT Handle
  • >50,000 MAU (need paid tier upgrades)
  • >500 concurrent users (Vercel limit)
  • Live ball-by-ball scoring
  • Real-time playing XI auto-detection
  • Automated payment/billing
  • >500 MB database (need Supabase upgrade)
  • >10,000 Redis commands/day (need Upstash upgrade)
  • Custom analytics per user
  • Player photo hosting
  • Multi-language content
These limitations are ACCEPTABLE for MVP. They become upgrade triggers when reached.
Section 11

MVP Quality Bar

Works Correctly
  • Form scores mathematically correct (10 player manual check)
  • Matchup data matches CricSheet source (5 pairs)
  • Venue stats match known records (3 major venues)
  • Dream team respects all constraints (role counts, balance)
  • API auth works (valid → success, invalid → 401)
  • Rate limiting works (101st call → 429)
  • No data shown for wrong match or player
Doesn't Crash
  • No 500 errors on any valid API call
  • No blank pages on any dashboard route
  • No JavaScript errors in browser console
  • Edge cases return graceful responses, not crashes
  • Admin panel doesn't corrupt data
  • Cron job failure doesn't break the product
Looks Acceptable
  • Landing page looks professional (can't look amateur)
  • Match detail page is readable and organized
  • Mobile screens are USABLE (not just "responsive")
  • Text is readable (font size, contrast)
  • Cards and tables are aligned
  • Loading states shown (not blank screens)
Fast Enough
  • API: <500ms response (p95)
  • Dashboard pages: <3s load (acceptable for MVP)
  • Admin recomputation: <90 seconds
  • No operation takes >5 minutes
Does NOT Need
  • Pixel-perfect design or animations
  • Empty state illustrations
  • Onboarding tutorial
  • Loading skeletons (spinners are fine)
  • Toast notifications
  • Perfect SEO or polished error pages
  • Analytics/tracking
RULE: If it works correctly, doesn't crash, looks acceptable, and is fast enough → IT SHIPS.
Section 12

Risks Specific to MVP

Risk 1Medium
Data Load Takes Too Long
CricSheet download slow or parser fails at scale. Impact: delays launch by 1–3 days.
Start data processing Day 1. If slow, use IPL + T20I subset first and add other leagues post-launch.
Risk 2Medium
Form Algorithm Produces Nonsensical Scores
Users lose trust immediately if Kohli shows form 2.1 or a tail-ender shows 9.5.
Manual verify 10 known players before launch. If scores wrong for Kohli/Bumrah level: DO NOT LAUNCH until fixed.
Risk 3Low–Med
No Active T20 League During Build
Can't test with real match data. Reduced launch demand.
Launch during ANY active T20 league. If none active, use recent completed matches as demo data. Wait for next league.
Risk 4Low
Free Tier Limits Hit During Testing
Supabase 500MB or Vercel 100K/month hit during development.
Monitor during development. Use local development (SQLite) more heavily to preserve production quota.
Risk 5Low
Operator Burns Out on Match Days
Match-day updates slip, data goes stale, users lose trust.
5 min/match is very manageable. Set phone alarm 1 hour before each match. Predicted XI fallback if truly unavailable.
Risk 6Medium — Biggest Risk
Nobody Signs Up
No validation, product built in a vacuum. The single biggest real risk.
Invite 10–20 known fantasy players personally. Post on r/Cricket, r/fantasycricket. DM 2–3 creators. If zero signups after 2 weeks → reassess distribution, not product. DON'T wait for organic traffic — actively distribute.
Risk 7Medium
Algorithm Accuracy Is Poor
First version always has issues. Users may lose trust before we can iterate.
Track accuracy from Day 1. Log every captain pick vs actual points. Iterate weekly. Be transparent: "v1 algorithm, improving with data." Users forgive early mistakes if you visibly improve.
